# Stratus Console (EC2 Replica) — সম্ভাব্য প্রশ্ন ও উত্তর

শিক্ষক যেসব প্রশ্ন করতে পারেন, এবং প্রতিটির উত্তরে **আসল AWS EC2** কীভাবে কাজ করে
বনাম **আমাদের প্রজেক্টে** কীভাবে করেছি — তার তুলনা।

---

## ১. VM আসলে কে তৈরি করে? AWS EC2 আর তোমাদেরটার মধ্যে পার্থক্য কী?

**AWS EC2:**
AWS-এর ডেটাসেন্টারে হাজার হাজার ফিজিক্যাল সার্ভার (host) আছে। প্রতিটি সার্ভারে একটি
**hypervisor** চলে। আগে AWS **Xen** ব্যবহার করত, এখন আধুনিক ইনস্ট্যান্সে **Nitro System**
(KVM-ভিত্তিক) ব্যবহার করে। EC2-এর **control plane** (API) সিদ্ধান্ত নেয় কোন ফিজিক্যাল
হোস্টে VM বসবে, তারপর hypervisor-কে VM তৈরি করতে বলে। ইউজার শুধু API কল করে।

**আমাদের প্রজেক্ট:**
আমার ল্যাপটপই "ডেটাসেন্টার"। **Multipass** (ভেতরে Linux-এ **QEMU/KVM** ব্যবহার করে)
আসল Ubuntu VM তৈরি করে। আমাদের Express ব্যাকএন্ড হলো **control plane** — এটি
`child_process` দিয়ে হোস্টের `multipass launch` কমান্ড চালায়।

**মূল কথা:** ধারণা একই (control plane → hypervisor → VM), শুধু স্কেল আলাদা।
AWS-এর control plane distributed, আমাদেরটা একটি Node.js সার্ভার।

## ২. ব্যাকএন্ড কেন VM-এর ভেতরে না রেখে হোস্টে চালাতে হলো?

এটা প্রজেক্টের সবচেয়ে গুরুত্বপূর্ণ আর্কিটেকচারাল সিদ্ধান্ত।

শুরুতে ব্যাকএন্ড একটা VM-এর (`my-local-ec2`) ভেতরে চলত। কিন্তু যখন **আসল VM তৈরি**
করার দরকার হলো, তখন সমস্যা হলো:
- `multipass` CLI শুধু **হোস্ট মেশিনে** ইনস্টল করা থাকে, VM-এর ভেতরে নয়।
- VM কখনোই হোস্টের `multipassd` ডেমনের সকেটে পৌঁছাতে পারে না।

তাই VM-এর ভেতরের ব্যাকএন্ড কখনোই আরেকটা VM বানাতে পারত না।

**AWS-এর সাথে মিল:** AWS-এও একটা EC2 ইনস্ট্যান্স নিজে নিজে হাইপারভাইজারকে বলে
আরেকটা ইনস্ট্যান্স বানাতে পারে না। **যাকে ম্যানেজ করবেন, তার বাইরে থেকেই করতে হয়।**
তাই আমার ল্যাপটপ = ক্লাউড প্রোভাইডার, আর VM-গুলো = EC2 ইনস্ট্যান্স।

## ৩. Key Pair কীভাবে কাজ করে? Key জেনারেট করার সময় পর্দার আড়ালে ঠিক কী ঘটে?

**AWS EC2:**
- ইনস্ট্যান্স তৈরির সময় আপনি একটি key pair বেছে নেন (RSA বা ED25519)।
- AWS **শুধু public key** নিজের কাছে রাখে; **private key (`.pem`) আপনি একবারই
  ডাউনলোড করতে পারেন** — AWS সেটা সেভ করে না। হারালে আর ফেরত পাওয়া যায় না।
- বুট করার সময় **cloud-init** ইনস্ট্যান্সের ভেতরে public key-টি
  `/home/ec2-user/.ssh/authorized_keys` (বা `ubuntu`) ফাইলে বসিয়ে দেয়।
- আপনি `ssh -i key.pem` দিয়ে কানেক্ট করলে SSH একটা **challenge–response** করে:
  সার্ভার একটা র‍্যান্ডম চ্যালেঞ্জ পাঠায়, ক্লায়েন্ট private key দিয়ে **সাইন** করে,
  সার্ভার authorized_keys-এ থাকা public key দিয়ে সেই সিগনেচার **ভেরিফাই** করে।
  **private key কখনো নেটওয়ার্কে যায় না।**

**আমাদের প্রজেক্ট (হুবহু একই ধারণা):**
1. প্রতিটি ইনস্ট্যান্সের জন্য Node-এর `crypto.generateKeyPairSync("rsa", {modulusLength: 2048})`
   দিয়ে **নতুন, ইউনিক** key pair বানাই।
2. Private key **PKCS#1 PEM** ফরম্যাটে এক্সপোর্ট করি (`-----BEGIN RSA PRIVATE KEY-----`)
   — ঠিক AWS-এর `.pem` স্টাইল।
3. Public key-কে **OpenSSH authorized_keys ফরম্যাটে** (`ssh-rsa AAAA...`) রূপান্তর করি।
   (RSA-র modulus `n` আর exponent `e` নিয়ে SSH wire-format এ এনকোড করে base64 করা হয়।)
4. VM লঞ্চের সময় **cloud-init**-এর `ssh_authorized_keys` দিয়ে public key ভেতরে বসিয়ে দিই।
5. Private key API রেসপন্সে **ঠিক একবার** পাঠাই, ব্রাউজার সেটি `<name>-key.pem` নামে
   ডাউনলোড করে, আর সার্ভার সেটি **কোথাও সেভ করে না**।

**অর্থাৎ:** নিরাপত্তার মডেল AWS-এর সাথে এক — সার্ভারে শুধু public key, private key
ইউজারের কাছে একবারই।

## ৪. IP অ্যাড্রেস কীভাবে পায়? AWS-এর VPC-র সাথে তুলনা করো।

**AWS EC2:**
- প্রতিটি ইনস্ট্যান্স একটি **VPC**-র **subnet**-এ থাকে এবং subnet-এর CIDR থেকে
  **private IPv4** পায় (AWS-এর DHCP দেয়)।
- **Public IP** আসলে ইনস্ট্যান্সের OS-এ থাকে না! VPC একটি **1:1 NAT** করে
  public IP → private IP ম্যাপ করে। ভেতরে `ifconfig` করলে শুধু private IP দেখবেন।
- ইনস্ট্যান্স **stop/start** করলে public IP **বদলে যায়** — স্থির রাখতে হলে
  **Elastic IP** নিতে হয়।

**আমাদের প্রজেক্ট:**
- Multipass একটি **ভার্চুয়াল ব্রিজ** (`mpqemubr0`, সাবনেট `10.176.164.0/24`) তৈরি করে।
- প্রতিটি VM ওই ব্রিজের DHCP থেকে একটি IP পায় (যেমন `10.176.164.147`)।
- এটা AWS-এর **private IP**-র সমতুল্য — ল্যাপটপের ভেতর থেকে রিচেবল, বাইরের ইন্টারনেট থেকে নয়।
- **আমাদেরও একই সমস্যা:** VM রিস্টার্ট করলে DHCP নতুন IP দিতে পারে। তাই আমরা
  **Start করার পর `multipass info` দিয়ে IP আবার পড়ে DB আপডেট করি।**

**সীমাবদ্ধতা (সৎভাবে):** আমাদের কোনো public IP / NAT / Elastic IP নেই।
সবই private-network লেভেলে।

## ৫. Instance Type (t2.micro / t2.medium) কীভাবে কাজ করে?

**AWS EC2:**
Instance type ঠিক করে কত vCPU, কত RAM, কেমন নেটওয়ার্ক ও স্টোরেজ পারফরম্যান্স
পাবেন (যেমন `t2.micro` = 1 vCPU, 1 GiB)। `t` ফ্যামিলি **burstable** — CPU credit
জমা করে, দরকারে বেশি CPU ব্যবহার করে। AWS এটা hypervisor লেভেলে enforce করে।

**আমাদের প্রজেক্ট:**
সার্ভারে একটি **catalog** রাখা আছে:
| Type | vCPU | RAM |
|------|------|-----|
| t2.micro | 1 | 1G |
| t2.small | 1 | 2G |
| t2.medium | 2 | 4G |

ইউজার ড্রপডাউন থেকে বেছে নিলে ব্যাকএন্ড সেটাকে বাস্তব কমান্ডে রূপ দেয়:
`multipass launch --cpus 2 --memory 4G ...` — QEMU/KVM সত্যিই ওই স্পেকে VM বানায়।

**নিরাপত্তা:** ক্যাটালগটাই **authoritative**। কেউ যদি API-তে `t2.micro` চেয়ে চুপিচুপি
`cpu: 64` পাঠায়, সার্ভার সেটা উপেক্ষা করে 1 vCPU-ই দেয় (টেস্ট করে দেখানো হয়েছে)।

**পার্থক্য:** AWS-এর burstable/CPU-credit মডেল আমাদের নেই — আমরা fixed vCPU দিই।

## ৬. ব্রাউজার থেকে টার্মিনাল (Instance Connect) কীভাবে কাজ করে?

**AWS EC2:**
দুইভাবে —
- **EC2 Instance Connect:** ব্রাউজার থেকে কানেক্ট করলে AWS একটি **অস্থায়ী public key**
  (মাত্র ~৬০ সেকেন্ডের জন্য) ইনস্ট্যান্সে পুশ করে, তারপর SSH করে। কাজ শেষে key মুছে যায়।
- **Session Manager (SSM):** ইনস্ট্যান্সে একটি **agent** চলে যা AWS-এর সাথে
  outbound connection রাখে — এতে SSH পোর্ট খোলাও লাগে না।

**আমাদের প্রজেক্ট:**
- ব্যাকএন্ড **`node-pty`** দিয়ে একটি সত্যিকারের **PTY (pseudo-terminal)** খোলে:
  `multipass exec <instance> -- bash -l`
- সেই টার্মিনালের input/output **WebSocket** (`/api/ws/terminal/:id`) দিয়ে
  ব্রাউজারে **xterm.js**-এর সাথে দুই দিকেই পাইপ করা হয়।
- ফলে ব্রাউজারেই আসল shell — prompt, রঙ, `top`, `vim` সব কাজ করে।

**গুরুত্বপূর্ণ পয়েন্ট:** আমরা এখানে **SSH ব্যবহার করি না**। কারণ ব্যাকএন্ড আগে থেকেই
হোস্টে আছে এবং Multipass-এর হাইপারভাইজার চ্যানেল ব্যবহার করতে পারে — অনেকটা
AWS-এর **SSM Session Manager**-এর মতো (SSH পোর্ট ছাড়াই কানেক্ট)। ডাউনলোড করা
`.pem` key তখনই দরকার যখন ইউজার নিজের টার্মিনাল থেকে `ssh -i key.pem` করতে চায়।

## ৭. Monitoring / CloudWatch কীভাবে করেছ? ডেটা কি আসল?

**AWS CloudWatch:**
- হাইপারভাইজার লেভেল থেকে মেট্রিক নেয়: `CPUUtilization`, `NetworkIn/Out`,
  `DiskReadBytes/WriteBytes` ইত্যাদি।
- ডিফল্টে **৫ মিনিট** ইন্টারভাল (detailed monitoring-এ ১ মিনিট)।
- **মজার তথ্য:** AWS ডিফল্টে **memory usage দেয় না!** কারণ হাইপারভাইজার VM-এর
  ভেতরের RAM ব্যবহার দেখতে পায় না — এর জন্য ভেতরে **CloudWatch Agent** বসাতে হয়।

**আমাদের প্রজেক্ট:**
- `GET /api/projects/:id/metrics` endpoint প্রতি ২ সেকেন্ডে পোল করা হয়।
- **আসল ডেটা:** memory usage, disk usage, এবং load average — সরাসরি
  `multipass info --format json` থেকে।
- **সিমুলেটেড ডেটা:** Network In/Out এবং Disk Read/Write — একটি smooth
  **bounded random walk** দিয়ে তৈরি (কারণ Multipass এগুলো দেয় না)।
- ফ্রন্টএন্ডে **Recharts** দিয়ে রিয়েল-টাইম লাইন চার্ট।

**সৎ স্বীকারোক্তি:** নেটওয়ার্ক/ডিস্ক I/O গ্রাফ সিমুলেটেড — এটা ডেমোর জন্য।
মজার ব্যাপার, **memory-র ক্ষেত্রে আমরা AWS-এর ডিফল্টের চেয়ে ভালো**, কারণ
Multipass আমাদের সরাসরি RAM ব্যবহার জানিয়ে দেয়।

## ৮. Start / Stop / Terminate-এ আসলে কী ঘটে?

**AWS EC2:**
| অ্যাকশন | কী হয় |
|---------|--------|
| **Stop** | VM বন্ধ, কিন্তু **EBS ভলিউম থাকে** (ডেটা থাকে)। কম্পিউটের বিল বন্ধ, স্টোরেজের বিল চলে। Public IP **হারিয়ে যায়**। |
| **Start** | আবার (সম্ভবত অন্য ফিজিক্যাল হোস্টে) চালু হয়, **নতুন public IP** পায়। |
| **Terminate** | VM ও (সাধারণত) root volume **স্থায়ীভাবে মুছে যায়** — ফেরত আসে না। |

State: `pending → running → stopping → stopped → shutting-down → terminated`

**আমাদের প্রজেক্ট (একদম একই সেমান্টিক্স):**
| অ্যাকশন | আসল কমান্ড |
|---------|------------|
| **Stop** | `multipass stop <name>` → ডিস্ক থাকে, DB-তে status `STOPPED` |
| **Start** | `multipass start <name>` → **IP আবার পড়ে DB আপডেট করি** (AWS-এর মতোই IP বদলাতে পারে) |
| **Terminate** | `multipass delete <name>` + `multipass purge` → VM চিরতরে মুছে যায়, DB থেকেও রেকর্ড ডিলিট |

`purge` না করলে multipass-এ VM "deleted" অবস্থায় পড়ে থাকে — তাই দুইটা কমান্ডই দরকার।

## ৯. ইনস্ট্যান্সের তথ্য কোথায় রাখো? AWS-এর সাথে তুলনা?

**AWS EC2:**
EC2-র **control plane** একটি বিশাল distributed ডেটাবেসে ইনস্ট্যান্সের সব
metadata রাখে (id, type, state, IP, key name, tags…)। `DescribeInstances` API
সেখান থেকেই পড়ে। হাইপারভাইজার আর ডেটাবেস — দুটো আলাদা লেয়ার।

**আমাদের প্রজেক্ট:**
- **Neon (ক্লাউড PostgreSQL)** + **Prisma ORM** আমাদের control-plane ডেটাবেস।
- `CloudProject` টেবিলে রাখা হয়: `id, name, description, status, instanceName,
  ipAddress, keyName, cpu, memory, instanceType, createdAt`।

**গুরুত্বপূর্ণ ধারণা:** ডেটাবেস হলো **desired/known state**, আর Multipass হলো
**actual state**। AWS-এও এই দুটো আলাদা। (আমাদের সীমাবদ্ধতা: কেউ যদি টার্মিনাল
থেকে সরাসরি `multipass delete` করে, DB-তে drift তৈরি হবে — AWS-এ reconciliation
লুপ থাকে, আমাদের নেই।)

---

## ১০. কোন কোন ফিচার AWS-এ আছে কিন্তু তোমাদের নেই? (সীমাবদ্ধতা)

সৎভাবে বলা ভালো — এগুলোই "future work":

| AWS EC2 ফিচার | আমাদের অবস্থা |
|----------------|----------------|
| **Security Group** (stateful firewall) | নেই — কোনো পোর্ট ফিল্টারিং করি না |
| **VPC / Subnet / Route Table** | নেই — Multipass-এর একটাই ফ্ল্যাট ব্রিজ নেটওয়ার্ক |
| **Elastic IP** (স্থির IP) | নেই — DHCP-তে IP বদলাতে পারে |
| **AMI নির্বাচন** | নেই — সবসময় ডিফল্ট Ubuntu ইমেজ |
| **EBS ভলিউম / Snapshot** | নেই — আলাদা স্টোরেজ ম্যানেজমেন্ট নেই |
| **Auto Scaling / Load Balancer** | নেই |
| **IAM / ইউজার লগইন** | নেই — যে কেউ ড্যাশবোর্ড খুললেই কন্ট্রোল করতে পারবে |
| **HTTPS** | নেই — লোকালহোস্টে চলে, তাই private key HTTP-তে যায় |
| **Billing / Metering** | নেই |

**তবে যা আছে তা আসল:** VM তৈরি, স্টার্ট/স্টপ/টার্মিনেট, key pair, ব্রাউজার টার্মিনাল,
মনিটরিং — সবই সত্যিকারের VM-এর উপর কাজ করে, কোনো mock নয়।
